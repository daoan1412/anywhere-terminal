# Research: shiki-v3-api

## 1. Shiki v4 static-only bundle construction

### Answers

- The current fine-grained core entry point is `createHighlighterCore` from `shiki/core`.
- The recommended JavaScript regex engine import for a browser/IIFE bundle is `createJavaScriptRegexEngine` from `shiki/engine/javascript`.
- The direct engine package `@shikijs/engine-javascript` is equivalent, but the `shiki/engine/javascript` re-export is the documented path and keeps the source import surface consistent.
- For static imports, language grammars come from `@shikijs/langs/<language-id>` and themes come from `@shikijs/themes/<theme-id>`.
- `light-plus` and `dark-plus` are available theme IDs in `@shikijs/themes`.
- Latest stable package versions as of 2026-05:
  - `shiki`: `4.1.0`
  - `@shikijs/langs`: `4.1.0`
  - `@shikijs/themes`: `4.1.0`
  - `@shikijs/engine-javascript`: `4.1.0`

### Exact signatures

```ts
export async function createHighlighterCore(
  options: HighlighterCoreOptions<false>
): Promise<HighlighterCore>
```

```ts
export function createJavaScriptRegexEngine(
  options: JavaScriptRegexEngineOptions = {}
): RegexEngine
```

`createJavaScriptRegexEngine()` takes no arguments in the common case; the options object is optional.

### Complete working sample for one language + one theme

```ts
import { createHighlighterCore } from "shiki/core";
import { createJavaScriptRegexEngine } from "shiki/engine/javascript";

import typescript from "@shikijs/langs/typescript";
import githubLight from "@shikijs/themes/github-light";

const highlighter = await createHighlighterCore({
  engine: createJavaScriptRegexEngine(),
  langs: [typescript],
  themes: [githubLight],
});
```

### Recommended approach

Use static ESM imports for exactly the grammars/themes you need and pass those modules to `createHighlighterCore`. That is the right shape for an esbuild-produced IIFE bundle because it avoids runtime `import()` and avoids shipping unused languages/themes.

## 2. `@shikijs/markdown-it` integration

### Answers

- `@shikijs/markdown-it` has two relevant modes:
  - the default export auto-creates its own highlighter
  - `fromHighlighter` in `@shikijs/markdown-it/core` consumes a pre-built highlighter
- For a static-only bundle, use `fromHighlighter`; do not rely on the default export if the goal is to avoid hidden bundle/runtime loading.
- The plugin does not switch themes by calling into Shiki per render in a JS callback; it emits multi-theme HTML using CSS variables, and the browser decides which theme wins via CSS.
- Theme switching is therefore initialized once at plugin setup time, then applied at render time through generated CSS variable markup.
- The `themes` option uses a mapping object, typically `{ light: 'github-light', dark: 'github-dark' }`.
- `defaultColor` controls which theme gets the inline `color` style; default is `light`.

### Exact `fromHighlighter` signature and setup shape

```ts
export function fromHighlighter(
  highlighter: HighlighterGeneric<any, any>,
  options: MarkdownItShikiSetupOptions,
) {
  return function (markdownit: MarkdownIt) {
    setupMarkdownIt(markdownit, highlighter, options)
  }
}
```

`MarkdownItShikiSetupOptions` is the Shiki setup/options shape extended with markdown-it extras; the important parts for this project are the `themes` mapping, `defaultColor`, `cssVariablePrefix`, `defaultLanguage`, and `fallbackLanguage`.

### Complete working sample with preloaded highlighter + light/dark themes

```ts
import MarkdownIt from "markdown-it";
import { fromHighlighter } from "@shikijs/markdown-it/core";
import { createHighlighterCore } from "shiki/core";
import { createJavaScriptRegexEngine } from "shiki/engine/javascript";

import typescript from "@shikijs/langs/typescript";
import javascript from "@shikijs/langs/javascript";
import githubLight from "@shikijs/themes/github-light";
import githubDark from "@shikijs/themes/github-dark";

const highlighter = await createHighlighterCore({
  engine: createJavaScriptRegexEngine(),
  langs: [typescript, javascript],
  themes: [githubLight, githubDark],
});

const md = new MarkdownIt({ html: false, linkify: false });
md.validateLink = () => false;
md.use(
  fromHighlighter(highlighter, {
    themes: {
      light: "github-light",
      dark: "github-dark",
    },
    defaultColor: "light",
    defaultLanguage: "text",
  }),
);
```

### Can it share grammars and avoid duplicate bundling?

Yes. The key is to build one highlighter instance with the exact imported language/theme modules and reuse that same instance with `fromHighlighter`. That avoids the default export’s auto-created highlighter path and keeps the bundle under your control.

### Recommended approach

Use `fromHighlighter` from `@shikijs/markdown-it/core` plus one shared `createHighlighterCore` instance. That is the only variant that cleanly fits a static-only IIFE bundle and lets you reuse the same loaded grammars/themes across any other rendering path.

## 3. VS Code `languageId` from a URI

### Answers

- Best fit for a hot-path hover preview: a small hand-maintained extension-to-`languageId` map for the curated 20 grammars.
- Why this wins: zero file read, zero document creation, no language service activation, and the fastest possible lookup.
- Downsides: it does not automatically respect user `files.associations` overrides unless you explicitly add your own config overlay.
- `openTextDocument(uri)` is the correctness-first option, but it reads the file and creates a text document just to get `document.languageId`.
- Inspecting `vscode.extensions.all` and matching `contributes.languages.extensions` is faster than opening the file, but still misses user associations and adds manifest-scanning complexity.
- There is no single public VS Code API that returns the fully effective `languageId` for an arbitrary URI without opening or resolving the document yourself.
- A useful extra public input is `vscode.workspace.getConfiguration('files').get('associations')`, which can be layered on top of the curated map if user overrides matter.

### Comparison

| Approach | Pros | Cons | Verdict |
| --- | --- | --- | --- |
| a) `openTextDocument(uri).then(doc => doc.languageId)` | Correct, respects VS Code’s document language resolution | Reads the file again; creates a document model; heavier for hover hot paths | Good fallback, not ideal here |
| b) Scan `vscode.extensions.all` and match `contributes.languages.extensions` | No file read; can be fast if cached | Misses `files.associations`; still more work than a map; relies on extension manifests rather than effective language mode | Better than openTextDocument for some cases, but not the best fit |
| c) Curated extension → `languageId` map | Fastest; no file read; no activation; simplest runtime behavior | Must be maintained; misses custom user associations unless you add a config overlay | Recommended |
| d) Other public API | No single API returns the effective `languageId` for a URI without opening a document | N/A | No better public API found |

### Code sample for the recommended approach

```ts
import * as vscode from "vscode";

const LANGUAGE_BY_EXTENSION: Record<string, string> = {
  ".ts": "typescript",
  ".tsx": "tsx",
  ".js": "javascript",
  ".jsx": "jsx",
  ".json": "json",
  ".yml": "yaml",
  ".yaml": "yaml",
  ".html": "html",
  ".css": "css",
  ".md": "markdown",
  ".py": "python",
  ".go": "go",
  ".rs": "rust",
  ".c": "c",
  ".cpp": "cpp",
  ".cc": "cpp",
  ".java": "java",
  ".kt": "kotlin",
  ".kts": "kotlin",
  ".sh": "shellscript",
  ".sql": "sql",
  ".rb": "ruby",
  ".php": "php",
};

function languageIdFromUri(uri: vscode.Uri): string {
  const match = uri.path.toLowerCase().match(/\.[^.]+$/);
  return (match && LANGUAGE_BY_EXTENSION[match[0]]) || "plaintext";
}
```

If user-defined file associations must be honored, layer `files.associations` config on top of this map instead of opening the file for every hover.

### Recommended approach

Use the curated map as the fast path, optionally augmented with `files.associations` from user/workspace settings. Do not use `openTextDocument` for every hover unless correctness beats latency and allocation cost.

## 4. esbuild + IIFE + Shiki bundle behavior

### Answers

- Static ESM imports of only the needed `@shikijs/langs/*` and `@shikijs/themes/*` modules do tree-shake cleanly in esbuild as long as those imports are unused by your code.
- If you statically import a grammar/theme and pass it into the highlighter config, esbuild will keep it; tree-shaking only removes imports that are proven unused.
- The fine-grained Shiki docs recommend ESM-based fine-grained imports over the full bundle for web use. They also emphasize dynamic imports for chunk splitting, but static imports are still valid when you intentionally want one IIFE.
- For this setup, the important property is “only import what you want”; do not import the full `shiki`, `shiki/bundle/full`, or `shiki/bundle/web` packages.
- Directional size estimate: yes, a 20-grammar + 4-theme static bundle with the JavaScript regex engine should plausibly fit under ~1.3 MB minified, and likely much lower than that. The exact number depends on which grammars are chosen and whether you keep only the curated set.
- Shiki’s own published reference points are much larger for the full bundle (`shiki/bundle/web`), while `shiki/core` is small; that is the right baseline for why a fine-grained bundle should stay comparatively compact.

### Code sample showing the bundle shape

```ts
import { createHighlighterCore } from "shiki/core";
import { createJavaScriptRegexEngine } from "shiki/engine/javascript";
import typescript from "@shikijs/langs/typescript";
import javascript from "@shikijs/langs/javascript";
import githubLight from "@shikijs/themes/github-light";
import githubDark from "@shikijs/themes/github-dark";

export const highlighterPromise = createHighlighterCore({
  engine: createJavaScriptRegexEngine(),
  langs: [typescript, javascript],
  themes: [githubLight, githubDark],
});
```

### Practical guidance for the IIFE build

- Keep the import list explicit and finite.
- Do not use `import()` or any runtime fetch for grammars/themes.
- Verify the final size with esbuild’s metafile/analyzer rather than guessing; the bundle size will be dominated by the exact grammar set, not by the highlighter API itself.

### Recommended approach

Use the static-import fine-grained Shiki path. esbuild can tree-shake unused modules, but because you intentionally reference the imported grammars/themes, the bundle will include exactly those modules and nothing more.

# Research: xterm.js 6.x Click-to-Cursor Positioning

## Answers

- **Can click coordinates be mapped to cell coordinates?** Yes — via `terminal.dimensions.css.cell` (public API) for manual calculation, or via internal `IMouseService.getCoords()` (not public). The calculation is: `col = floor((clientX - rect.left) / cssCellWidth)`, `row = floor((clientY - rect.top) / cssCellHeight)`, clamped to `terminal.cols`/`terminal.rows`.
- **Is there a built-in click-to-cursor feature?** Yes — `altClickMovesCursor: true` (default `true` in xterm.js) handles Alt+click cursor repositioning via escape sequences. This is the **recommended starting point**.
- **How does xterm.js move the shell cursor?** Sends CSI escape sequences (Cursor Forward/Backward/Up/Down) via the `onData` event. The sequences are generated by `moveToCellSequence()` in `src/browser/input/MoveToCell.ts`. These are relative movements, not absolute positioning, to work across normal and alternate buffers.
- **What is the public API for sending input?** `terminal.input(data: string, wasUserInput?: boolean)` — sends data as if typed. `terminal.write(data)` sends output (not input). In a VS Code extension, the `onData` handler must forward data to the PTY.
- **Are the mouse service APIs public?** No. `IMouseService`, `moveToCellSequence`, and `_core` are internal. The `altClickMovesCursor` behavior is baked into `SelectionService` and not exposed as a callable function.

## Recommended Approach

1. **First, try enabling `altClickMovesCursor: true`** in the terminal options. This is already implemented in xterm.js and handles coordinate calculation + escape sequence generation automatically. The built-in behavior only activates with Alt key — but the underlying mechanism can be reused.
2. **For plain click (no modifier), implement custom handler:**
   - Listen to `mousedown`/`mouseup` on the terminal element
   - Convert pixel coordinates to cell coordinates using `terminal.dimensions.css.cell.width`/`height` and `element.getBoundingClientRect()`
   - Reuse the escape sequence logic from `moveToCellSequence` (copy the algorithm — it's MIT licensed)
   - Send the sequence via `terminal.input(escapeSequence)` or through the PTY's `onData` handler
3. **Alternative approach:** Instead of sending relative cursor movement sequences, send `CSI Ps ; Ps H` (CUP - Cursor Position) for absolute positioning. However, this only works correctly in the alternate buffer or when the shell supports absolute cursor positioning.

## Confidence

**High** — confirmed via official deepwiki docs (xtermjs/xterm.js), Microsoft/vscode source code showing real usage of `altClickMovesCursor`, and xterm.js source structure for `MoveToCell.ts` and `SelectionService`.

## Core API

### Coordinate Calculation (Manual, using public APIs)

```typescript
// terminal must be opened first; dimensions is undefined before open()
const dims = terminal.dimensions!;
const rect = terminal.element!.getBoundingClientRect();

function pixelToCell(clientX: number, clientY: number): { col: number; row: number } | null {
  const cssCellWidth = dims.css.cell.width;
  const cssCellHeight = dims.css.cell.height;

  const col = Math.floor((clientX - rect.left) / cssCellWidth);
  const row = Math.floor((clientY - rect.top) / cssCellHeight);

  if (col < 0 || col >= terminal.cols || row < 0 || row >= terminal.rows) {
    return null;
  }
  return { col, row };
}
```

### `terminal.input()` — Send user input

```typescript
// Sends escape sequences as if the user typed them
terminal.input('\x1b[100C'); // Move cursor forward 100 columns
```

### `terminal.onData()` — Listen for input to forward to PTY

```typescript
terminal.onData(data => {
  // Forward `data` to the PTY (in a VS Code extension, this goes to pty.write())
  pty.write(data);
});
```

## Escape Sequences for Cursor Movement

The `moveToCellSequence` function in xterm.js generates these sequences:

| Direction | Normal Mode | Application Mode |
|-----------|-------------|------------------|
| Up | `\x1b[{n}A` | `\x1b[{n}OA` |
| Down | `\x1b[{n}B` | `\x1b[{n}OB` |
| Forward (right) | `\x1b[{n}C` | `\x1b[{n}OC` |
| Backward (left) | `\x1b[{n}D` | `\x1b[{n}OD` |

**Absolute positioning (CUP):** `\x1b[{row};{col}H` — moves cursor directly to (row, col). 1-indexed.

**Character Absolute (CHA):** `\x1b[{col}G` — moves cursor to column on current row.

## How xterm.js altClickMovesCursor Works (Internal Flow)

1. `SelectionService._handleMouseUp()` intercepts mouseup event
2. Checks conditions:
   - `selectionText.length <= 1` (no active selection)
   - `timeElapsed < 500ms` (quick click, not drag)
   - `event.altKey` pressed
   - `altClickMovesCursor` option enabled
   - `ybase === ydisp` (not scrolled back)
3. Calls `_mouseService.getCoords()` to get (col, row)
4. Calls `moveToCellSequence(targetCol, targetRow, bufferService, applicationCursorKeys)`
5. Generates escape sequence string
6. Triggers `coreService.triggerDataEvent()` → fires `onData` event
7. Application (PTY) receives escape sequence → shell interprets it → cursor moves

### Normal Buffer Movement Logic

- Same row: horizontal movement only (`LEFT` or `RIGHT` sequences)
- Different rows: calculates total cell distance (accounting for line width) and moves with a single direction sequence

### Alternate Buffer Movement Logic

- 3-step: `resetStartingRow` → `moveToRequestedRow` → `moveToRequestedCol`
- Accounts for wrapped lines

## Platform-Specific Considerations

### VS Code Extension + Webview

- xterm.js runs in the webview's HTML/JS context
- The PTY runs in the extension host (Node.js process)
- Communication is via `postMessage` between webview and extension host
- The `onData` handler in the webview sends data to the extension host, which calls `pty.write(data)`
- Conversely, PTY output goes through `pty.onData` → `postMessage` → `terminal.write()`

### macOS

- `macOptionClickForcesSelection: true` forces Option+click to select text instead of moving cursor
- On macOS, the Option key maps to `altKey` in browser events

## Gotchas & Constraints

### Shell/Readline Limitations

- **Click-to-cursor only works in line-editing mode** (bash readline, zsh line editor). The escape sequences move the cursor within the current input buffer.
- **Does NOT work in fullscreen TUI apps** (vim, nano, htop, tmux) — these applications handle cursor positioning themselves and ignore relative movement sequences from the terminal input.
- **Does NOT work with multi-line input** in all shells — the escape sequence approach moves the visible cursor, but the shell's internal cursor position may differ for wrapped lines.
- **Shell must support ANSI escape sequences** — virtually all modern shells do, but some minimal environments (busybox sh, embedded systems) may not.

### xterm.js Limitations

- `terminal.dimensions` is `undefined` before `terminal.open()` — guard against this
- `terminal.dimensions` changes on resize — subscribe to `onDimensionsChange`
- Internal services (`IMouseService`, `moveToCellSequence`) are **not part of the public API** — copying the algorithm is the safest approach
- The built-in `altClickMovesCursor` requires the Alt key — there is no built-in plain-click equivalent

### Scrollback

- Cursor movement only works when `ybase === ydisp` (viewport at bottom). Clicking while scrolled back should be a no-op.
- The target row is relative to the visible viewport, not the absolute buffer position.

### Application Cursor Mode

- When `applicationCursorKeys` is true (e.g., in vim), escape sequences use `OA/OB/OC/OD` instead of `A/B/C/D`. The `moveToCellSequence` function accounts for this.

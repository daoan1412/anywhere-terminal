## Notes

**Triage (2026-05-24)**
- Complexity: **STANDARD**
- Escalation flags: `cross-boundary` (new extension↔webview RPC + new vscode.workspace.createFileSystemWatcher integration), MEDIUM risk
- Required artifacts: proposal + specs + design + tasks (full ceremony)
- Discovery: Full — finder workstreams for current file-tree architecture + IFileSystemProvider extension points; librarian workstream for VS Code FileSystemWatcher patterns (lifecycle, throttling, glob limits)

**Problem statement**
Pasting/renaming/deleting files in VS Code Explorer is not reflected in the AnyWhere Terminal file tree. The git decoration provider DOES detect changes (and emits "untracked" deltas correctly), but the new file path is not in `nodeCache` so the status is stashed in `pendingStatuses` and the directory is never re-read because no FS watcher exists. There is no `vscode.workspace.createFileSystemWatcher`, no `onDidCreateFiles/Delete/Rename` subscription anywhere in the file tree code path.

**Key constraint from user**
The file tree can browse ANY directory (not only the workspace root). The watcher set must be **dynamic** — bound to the directories the user actually has loaded/expanded, not statically to workspace folders. Watcher lifecycle must follow directory loading/unloading + collapse/eviction.

# Design: restore-terminal-sessions

## Decisions

### D1: Authoritative buffer state lives in an extension-host xterm-headless mirror, not in the webview

Per-session, on first PTY data, `SessionManager` lazily constructs `new HeadlessTerminal({cols, rows, scrollback: 1000, allowProposedApi: true})` from `@xterm/headless`. Every byte from `pty.onData` is forwarded to `headless.write(data)` (in addition to the existing webview forward). `resize(cols, rows)` is mirrored too. This replaces the raw `scrollbackCache: string[]` for restore purposes; the cache stays in the codebase only as a fast in-memory replay for L1 webview reload (single-line removal is invasive and the cache pays for itself there).

Snapshot is `session._serializeAddon.serialize({scrollback: 1000, excludeAltBuffer: true, excludeModes: true})`. Options match VS Code core (`ptyService.ts:1086-1089`).

**SerializeAddon instance cached per session**: dynamic `import('@xterm/addon-serialize')` is invoked at most once per session (the first time we need a snapshot for it). The resulting `SerializeAddon` instance is loaded into the session's headless terminal via `headless.loadAddon(addon)` and stored on the session. Subsequent snapshots reuse the same instance — no new `loadAddon` calls. The addon is disposed with the session.

**Rejected alternative**: webview-side `SerializeAddon` + `postMessage` snapshot (PLAN.md "simpler approach"). AT defers webview rendering for inactive tabs — a sidebar terminal that was never visible has no webview xterm to serialize from. VS Code core itself does not use webview-side serialization for restore.

**Rejected alternative**: create a fresh `SerializeAddon` instance per snapshot (VS Code core does this in `XtermSerializer.generateReplayEvent`). VS Code only serializes on detach (low frequency); AT serializes on a 1s debounce during hot output (high frequency) and so caches per-session to avoid churn.

### D2: Stable panel identity is webview-resident via `vscode.setState({panelId})`

`TerminalEditorProvider` accepts an optional `panelId: string` (UUID) in its constructor.

- **New panel** (`createPanel` factory): generate `panelId = crypto.randomUUID()`. After the webview signals `ready`, post a one-shot `setPanelId` message; the webview merges `{panelId}` into `vscode.setState(...)`.
- **Revived panel** (`TerminalPanelSerializer.deserializeWebviewPanel(panel, state)`): pass `state.panelId` through to the new `TerminalEditorProvider` instance.
- `_viewId = "editor-${panelId}"` — stable derivation. SessionManager's view-keyed maps (`viewSessions`) thus track the same key across reload.

**Rejected alternative**: assigning panelId from extension host via `workspaceState`. The webview is the natural carrier — VS Code persists `vscode.getState()` as part of webview revival automatically.

### D3: `onDidDispose` schedules destroy with a 5000 ms grace; serializer revival cancels it AND rebinds the surviving sessions to the new webview

`SessionManager.scheduleDestroyForView(viewId, delayMs = 5000)` registers a `setTimeout` keyed by `viewId` in `pendingDestroys: Map<string, NodeJS.Timeout>`. On fire: `destroyAllForView(viewId)`. `cancelScheduledDestroy(viewId)` clears the timer.

`TerminalEditorProvider.onDidDispose` is rewritten to call `scheduleDestroyForView(this._viewId, READ_SETTING_OR_DEFAULT)`. The serializer's `deserializeWebviewPanel` calls `cancelScheduledDestroy(viewId)` BEFORE creating the new `TerminalEditorProvider`. After reload the serializer typically fires within ~1s (VS Code's main thread re-attaches webviews immediately).

**Webview rebind on revive (CRITICAL — fixes oracle finding Q2)**: when `TerminalEditorProvider.setupPanel()` finishes and the new webview signals `ready`, the provider MUST call `sessionManager.updateWebviewForView(viewId, this._panel.webview)` BEFORE replaying scrollback. Without this, the surviving sessions still hold the disposed webview reference and `safePostMessage` silently no-ops. The sidebar/panel already does this at `src/providers/TerminalViewProvider.ts:582-608` (existing `existingTabs` branch) — the editor needs the same call. This rebind happens regardless of whether snapshots are being consumed (Phase A grace-period revive) or whether sessions are being created from snapshots (Phase B cross-restart).

`SessionManager.dispose()` synchronously fires every pending timer's callback (`destroyAllForView`) so no PTYs leak when the extension host shuts down with grace timers in flight.

**Edge case — close-then-quick-reopen**: user closing a tab and opening a fresh one within the grace window. The new panel has a different `panelId` (generated by `createPanel`) and the serializer is not invoked (only window-reload triggers serializer per VS Code API). The 5s timer for the closed panel still fires correctly. Accepted; matches VS Code's own behavior.

**Edge case — back-to-back reloads**: user triggers Cmd+R twice in rapid succession. The first reload calls dispose → schedules a 5s timer. The serializer fires the first revival → cancels timer. The second reload disposes the just-created panel → schedules a fresh timer. The serializer fires the second revival → cancels again. Each cycle is idempotent because `cancelScheduledDestroy` removes the entry from `pendingDestroys`; `scheduleDestroyForView` overwrites any same-viewId entry (clear-then-set) so a second schedule never leaves an orphan timer.

### D4: Persistence schema — metadata index in `workspaceState`, serialized buffers in `context.storageUri`

**Why two-tier (fixes oracle MUST-FIX 1)**: VS Code warns when extension `workspaceState` exceeds 512KB per `src/vs/platform/extensionManagement/common/extensionStorage.ts:47` (`LARGE_STATE_WARNING_THRESHOLD`). The original plan stored 20×1MB buffers in `workspaceState` — explicitly anti-pattern. Buffers go to disk via `context.storageUri`; only metadata stays in `workspaceState`.

```ts
// workspaceState key: anywhereTerminal.editorPanels.live
interface LiveEditorPanelsRecord {
  version: 1;
  panels: Array<{
    panelId: string;
    sessionIds: string[];
    createdAt: number; // epoch ms
    updatedAt: number; // epoch ms
  }>;
}

// workspaceState key: anywhereTerminal.sessionSnapshots.index (metadata only)
interface SessionSnapshotsIndex {
  version: 1;
  entries: Record<string /* sessionId */, SessionSnapshotMetadata>;
}

interface SessionSnapshotMetadata {
  // Identity
  sessionId: string;
  panelId?: string;                                 // present when viewLocation === "editor"
  viewLocation: "sidebar" | "panel" | "editor";
  terminalNumber: number;
  customName: string | null;

  // PTY config (for respawn)
  shell: string;
  shellArgs: string[];
  cwd: string;                                       // initial cwd from spawn
  currentCwd: string | null;                         // latest OSC 7/633; null if never reported

  // Visual state — dimensions kept inline; buffer is a file pointer
  cols: number;
  rows: number;
  bufferFile: string;                                 // relative path under storageUri: "snapshots/<sessionId>.snapshot.ans"
  bufferBytes: number;                                // size of bufferFile in bytes; used for eviction caps

  // Split-pane (D12)
  isSplitPane: boolean;
  rootTabId: string | null;                           // sessionId of the tab root; null for non-split sessions; same as sessionId for the root of a split

  // Metadata
  snapshotAt: number;                                 // epoch ms
  shellExited: boolean;                               // restored read-only when true (D13)
  exitCode: number | null;                            // recorded when shell exits
}

// On-disk format under context.storageUri/snapshots/<sessionId>.snapshot.ans
// = raw UTF-8 string returned by SerializeAddon.serialize(...)
```

Buffer files are written via `fs/promises.writeFile` during debounced flush and `fs.writeFileSync` during the synchronous `deactivate()` path (D6). On read, missing/unreadable buffer files cause the matching index entry to be dropped silently (logged at debug level).

Eviction (D5) is applied on the index; orphan buffer files (in storage dir but not referenced) are deleted on hydrate. Steady-state disk footprint is bounded by `20 × 1 MB = 20 MB` under `storageUri`; the index in `workspaceState` is roughly `20 × ~400 bytes = ~8 KB` (well under the 512 KB warning).

### D5: Eviction policy

Applied during `hydrateFromSnapshots(record)` on activate AND during the debounced write:

1. Drop entries with `snapshotAt < Date.now() - 7 * 24 * 60 * 60 * 1000`.
2. Drop entries with `bufferBytes > 1_048_576` (defensive — write-side already truncates to 1 MB).
3. Keep the 20 most recent (sorted by `snapshotAt` desc); drop the rest.
4. On hydrate only: scan `storageUri/snapshots/` and delete any `.snapshot.ans` file not referenced by the surviving index entries (orphan cleanup).

Eviction during write runs BEFORE the `workspaceState.update` call AND before the buffer-file write. Dropped entries have their buffer files unlinked from disk in the same flush.

Write-side serialized-buffer truncation: if `serialize()` returns >1MB UTF-8 bytes, trim the leading bytes back to the nearest CR/LF boundary at ≤1MB. Truncation is best-effort — ANSI sequence integrity isn't guaranteed by simple bytewise trim, but xterm's parser is resilient to incomplete escape sequences at the start of a `write()` (skips unparseable bytes). The truncated length is the value persisted as `bufferBytes`.

### D6: Snapshot cadence — debounced on PTY data + sync flush on deactivate

No periodic timer. Triggers:

- **Debounced (1000 ms) on `pty.onData`** — coalesces hot loops. Implementation: `SessionManager.schedulePersist()` sets a single `setTimeout`; the timer flushes ALL pending snapshots for ALL sessions in one pass to minimize `workspaceState.update` calls.
- **Synchronous flush on `extension.deactivate`** — see explicit ordering below.
- **Debounced on session metadata mutations** (rename, cwd change): rides the next 1s tick.

**Synchronous deactivate flush (fixes oracle MUST-FIX 2)**:

VS Code's extension host has a ~6000ms shutdown grace per `src/vs/platform/extensions/common/extensionHostStarter.ts`. `ExtensionMemento.update()` is scheduled via `RunOnceScheduler` (0ms) then proxied to main thread storage per `src/vs/workbench/api/common/extHostMemento.ts:35-100`. Awaiting the returned `Thenable` is best-effort — host can be killed before the main-thread ack lands.

Two-step deactivate to maximize durability:

1. **Buffer files first, synchronously**: `SessionManager.flushSnapshotsSync()` (called from `deactivate` BEFORE `context.subscriptions.dispose()`) generates the serialized buffer for every session via the cached SerializeAddon and writes each file via `fs.writeFileSync(...)` to `storageUri/snapshots/<sessionId>.snapshot.ans`. Synchronous I/O blocks the event loop but is bounded (20 sessions × ~1MB = ~20MB write) and finishes well within the 6s budget. PTYs are NOT killed during this pass.
2. **Index second, awaited**: `await context.workspaceState.update(SESSION_SNAPSHOTS_INDEX_KEY, index)` and `await context.workspaceState.update(LIVE_EDITOR_PANELS_KEY, livePanels)`. Even if killed mid-await, the buffer files have already landed on disk; the index lost will be reconstructed from buffer files on next hydrate (orphan files are picked up as a fallback — see D5 step 4 modification).

`SessionManager` is REMOVED from `context.subscriptions` and instead disposed explicitly from `deactivate()` so the order is controlled. Disposal is idempotent (`_disposed` guard) — re-entry from a stray subscription call is a no-op.

**Hydrate fallback**: if `workspaceState` index is missing but buffer files exist (e.g. killed mid-await above), `hydrateFromSnapshots` reads each `.snapshot.ans` file and reconstructs a minimal index entry with defaults (`viewLocation` parsed from the parent panel record if present, else `"sidebar"`; missing `customName/currentCwd` set to null). This is best-effort partial recovery — better than dropping everything.

**Rejected alternative**: periodic 30s timer (PLAN.md). Adds a constant wake-up per session for no durability gain — debounced flush plus sync deactivate flush covers the loss window.

### D7: Activate-time hydrate flow + editor existing-session reload branch

```
extension.activate(context)
  ├── const index = context.workspaceState.get<SessionSnapshotsIndex>(SESSION_SNAPSHOTS_INDEX_KEY)
  ├── const livePanels = context.workspaceState.get<LiveEditorPanelsRecord>(LIVE_EDITOR_PANELS_KEY)
  ├── await sessionManager.hydrateFromSnapshots(index)   // reads buffer files, applies eviction, deletes orphans
  ├── sessionManager.hydrateLivePanels(livePanels)
  └── register providers + WebviewPanelSerializer
       │
       ├── TerminalViewProvider.resolveWebviewView(sidebar | panel)  // unchanged registration
       │     and onReady →
       │       ├── if (existingTabs.length > 0)
       │       │     → existing scrollback reload path (unchanged, sessionManager.updateWebviewForView already happens)
       │       └── else if (sessionManager.hasSnapshotsForLocation(loc))
       │             ├── const snaps = sessionManager.consumeSnapshotsForLocation(loc)
       │             ├── for each snap: createSession({restoreFrom: snap})   (creates fresh PTY if !shellExited; read-only ghost if shellExited)
       │             └── send init + restoreFromSnapshot messages
       │
       └── TerminalPanelSerializer.deserializeWebviewPanel(panel, state)
             ├── sessionManager.cancelScheduledDestroy(`editor-${state.panelId ?? legacyFallback}`)
             ├── const snaps = sessionManager.consumeSnapshotsForPanel(state.panelId)
             ├── new TerminalEditorProvider(..., panel, panelId: state.panelId ?? fresh, restoreSnapshots: snaps)
             └── setupPanel() →
                   onReady →
                     ├── const existingTabs = sessionManager.getTabsForView(this._viewId)  // NEW — checks viewId derived from panelId
                     ├── if (existingTabs.length > 0)
                     │     → sessionManager.updateWebviewForView(this._viewId, this._panel.webview)    (rebind — D3)
                     │     → for each tab: send "init" + send "restore" with scrollback cache
                     │     → DO NOT call createSession
                     ├── else if (restoreSnapshots.length > 0)
                     │     → for each snap: createSession({restoreFrom: snap})
                     │     → send init + restoreFromSnapshot
                     └── else (cold open from createPanel())
                           → existing path: createSession + init
```

The editor `onReady` branch above directly addresses oracle MUST-FIX 3 — currently `src/providers/TerminalEditorProvider.ts:532-549` always creates a new session, which would orphan PTYs that survived via the D3 grace period. The sidebar/panel symmetry is restored.

`consume*` methods remove the snapshot from the in-memory map after returning it — guarantees idempotent restore even if the provider is called twice.

**Legacy panels without `panelId` in state**: `state?.panelId` may be undefined for panels serialized by a prior version of AT (none deployed today, but accept as future-proofing). The serializer generates a fresh UUID, no snapshot is consumed (the index would have a different `panelId`), and the new panel comes up cold. Acceptable — no data loss because no prior version had snapshots to lose.

### D8: Restore message on the IPC channel + webview-side write order

New extension→webview message `restoreFromSnapshot`:

```ts
interface RestoreFromSnapshotMessage {
  type: "restoreFromSnapshot";
  tabId: string;                                    // sessionId
  serializedBuffer: string;
  cols: number;
  rows: number;
  snapshotAt: number;
}
```

Webview handler (in `TerminalFactory.createTerminal` extension or the `restore`/`init` orchestrator in `webview/main.ts`):

1. `terminal.resize(snapshot.cols, snapshot.rows)` BEFORE attaching to DOM, BEFORE writing buffer.
2. `terminal.write(snapshot.serializedBuffer)` — replay state.
3. `terminal.write(formatDivider(snapshot.snapshotAt))` — divider text (D9).
4. Attach to DOM via existing `open(container)` path; resume PTY forwarding via existing `data` message handler.
5. After attach, `fit()` to the actual pane dimensions (existing `FitAddon` flow).

Order matters: per xterm's `addons/addon-serialize/README.md`, writing serialized data into a non-`open()`'d terminal avoids wasted render cycles on partial frames.

### D9: Restore divider — exact UI string

```
\x1b[0m\r\n\x1b[2m─── restored — last update at HH:MM ───\x1b[0m\r\n
```

The leading `\x1b[0m` (SGR Reset) is REQUIRED (fixes oracle Q8): the serialized buffer can end with a non-default SGR state (e.g. bold/colored from the user's prompt), which would otherwise bleed into the divider line. The `\x1b[2m` after the reset re-applies dim, the trailing `\x1b[0m` cancels it. `HH:MM` is the local-time formatting of `snapshot.snapshotAt` using 24-hour zero-padded clock. Writing the divider is the webview's responsibility (D8 step 3) — the extension host doesn't need to inject it into the headless mirror.

**Exited-session divider variant (D13)**: when restoring a session with `shellExited === true`, append the exit marker on the next line:

```
\x1b[0m\r\n\x1b[2m─── restored — last update at HH:MM (shell exited, code: <N>) ───\x1b[0m\r\n
```

`<N>` is the recorded `exitCode` (or `?` when null).

### D10: Live editor panels tracking — append on create, evict on close, on dispose check serializer race

`workspaceState["anywhereTerminal.editorPanels.live"]` is the source of truth for "panels that existed at last persist." On `TerminalEditorProvider.createPanel`, append `{panelId, sessionIds: [], createdAt, updatedAt}`. On session create within a panel, push the sessionId. On real dispose (after grace expires without revival), remove the panel entry.

This list is what tells us, on a clean restart, which `panelId`s to expect deserializer calls for and which sessions belong to them. Without it, we'd need each individual session snapshot to know its panel — duplicated and harder to keep consistent. Tracking at the panel level scales O(panels) not O(sessions).

### D11: Setting — `anywhereTerminal.sessionRestore.enabled`

Single boolean, default `true`. Read once in `extension.activate` and passed to `SessionManager` constructor as `restoreEnabled: boolean`. When `false`:

- No headless mirror is constructed.
- `handleSnapshot` is a no-op.
- `hydrateFromSnapshots` clears any prior snapshots from `workspaceState` AND deletes the `storageUri/snapshots/` directory contents (so flipping the switch off cleans up disk).
- `WebviewPanelSerializer` is still registered (so panels still revive empty — no PTY orphaning), but `consumeSnapshotsForPanel` returns `[]` and the provider creates fresh sessions.

**Mid-session toggle to `false`**: the extension subscribes to `vscode.workspace.onDidChangeConfiguration`. When `sessionRestore.enabled` flips from `true` to `false`, `SessionManager`:
1. Disposes all headless mirrors + their cached SerializeAddon instances.
2. Cancels any pending debounced persist.
3. Calls `purge()` — deletes workspaceState index + buffer files.
4. Subsequent PTY output does NOT mirror, does NOT schedule persists.

The reverse toggle `false → true` lazily reconstructs the headless mirror on the next PTY output per session. No prior data is recoverable — accepted; user opted out then back in.

This is the kill switch for users who hit a regression; it also disables the new `@xterm/headless` runtime cost entirely.

### D12: Split-pane snapshot preservation

AT's split layout is owned by the WEBVIEW (`WebviewStateStore.tabLayouts: Map<string, SplitNode>`) and persisted via `vscode.setState` (`src/webview/state/WebviewStateStore.ts:197-247`). Sessions know whether they are split panes via `isSplitPane: boolean` on `TerminalSession`. The extension host does NOT own the split tree.

For cross-restart restore to work correctly with split panes, the index entry MUST carry enough information for the webview to reconstruct membership:

- `isSplitPane: boolean` — copied from the session at snapshot time.
- `rootTabId: string | null` — the session ID that owns the tab in the layout tree; equals `sessionId` for unsplit terminals and for the root of a split tree; differs only for child panes inside a split.

The webview already persists `tabLayouts` separately via `vscode.setState`. Both reload paths (Phase A grace revive and Phase B cross-restart) MUST follow the same order:

1. Extension calls `init` with ALL restored sessions (roots + split-pane children) and each `init.tabs[i]` carries `isSplitPane: boolean`. Without splits in `init.tabs`, the webview's `handleInit` builds `validTabIds` only from roots, and the layout restoration step (`WebviewStateStore.restore()` + the validation at `webview/main.ts:472-476`) discards any layout containing a split child — silently orphaning the PTYs in the extension host. **`SessionManager` exposes `getAllSessionsForView(viewId)` returning roots AND splits with the `isSplitPane` flag**; `getTabsForView` is preserved unchanged for callers that legitimately want roots only (tab strip, rename).
2. Webview's `handleInit` SHALL create xterm instances for every entry in `init.tabs`. For entries with `isSplitPane: true`, the webview MUST skip the side effects that `factory.createTerminal` applies to root tabs:
    - DO NOT initialize a per-pane `tabLayouts` leaf (the layout is owned by the root tab).
    - DO NOT set the pane id as `activeTabId`.
3. After every terminal is created, the webview SHALL call `splitRenderer.renderTabSplitTree(tabId)` for each `tabLayouts` entry whose root layout is a `branch`, then `splitRenderer.showTabContainer(tabId)` for the active root tab to make its multi-pane container visible.
4. For each session in restore order, the webview writes the snapshot buffer + divider into its xterm.

**Edge case — partial split**: snapshot has 2 of 3 split panes (the third evicted by caps). The webview's `handleInit` removes layouts whose root tab is not in `validTabIds`, but does NOT recursively prune dangling leaf references inside surviving layouts. Result: the missing leaf renders as a blank pane with no xterm attached (the leaf's `onLeafMounted` no-ops when `store.terminals.get(missingId)` is undefined). Accepted limitation — partial eviction is rare (caps drop the oldest entries; a tab and its split children are typically same-vintage). A future change MAY add recursive leaf pruning to `WebviewStateStore.restore()` or `handleInit`.

**No layout reconstruction in extension host**: the extension does NOT write a "layout" record. Layouts stay where they are today (`vscode.setState`).

### D13: Exited sessions restored read-only

When a session's shell exits naturally, `SessionManager` records the exit and stamps `shellExited: true` + `exitCode: number | null` on the next snapshot (and freezes further mirror updates).

On restore, `consume*` methods INCLUDE exited entries in the returned array (NOT dropped per oracle MUST-FIX 6). `createSession({restoreFrom: snap})` MUST honor `restoreFrom.shellExited === true`:

- Do NOT spawn a fresh PTY.
- Construct a `TerminalSession` with the persisted metadata; mark `isActive: false`.
- The session is registered with the view so it appears in the tab strip.
- The webview's `restoreFromSnapshot` writes buffer + the exited-variant divider (D9).
- The webview marks the `TerminalInstance` as `exited: true` (existing field — `src/webview/state/WebviewStateStore.ts:30-32`), suppressing input and reflecting the read-only style already used for exited terminals.
- User can still close the tab manually; the session is destroyed normally (no PTY to kill).

## Risk Map

| Component | Risk | Mitigation |
|---|---|---|
| `@xterm/headless` mirror per session | CPU overhead on hot PTY output | Mirror only when `restoreEnabled === true`. VS Code uses identical pattern at scale (`ptyService.ts:1067-1110`). Benchmark in tests; gate ship on <5% overhead at 100 KB/s PTY throughput. |
| `workspaceState` bloat | Persisted snapshots can grow unbounded | D5 caps: 1MB/snapshot, 20 snapshots, 7-day age. Eviction runs both on hydrate AND on write. |
| Extension-host crash before debounce flush | Up to 1s of buffer growth lost | Sync flush in `deactivate`; D6 documents the 1s loss window as accepted. |
| Editor panel `onDidDispose` ambiguity | Distinguishing reload vs close | D3 grace-period destroy + serializer-cancel. Pending timers fire if no revival; SessionManager.dispose synchronously fires all pending callbacks on extension shutdown so nothing leaks. |
| Restore vs new-session race in sidebar/panel | `existingTabs.length > 0` branch must take precedence | D7 explicitly guards new branch on `existingTabs.length === 0`. Existing `view-lifecycle-resilience` spec behavior is untouched. |
| `consumeSnapshotsForLocation` called twice | Double-restore if provider re-resolves | `consume*` removes from in-memory map on first call — idempotent. |
| Dynamic `import('@xterm/addon-serialize')` failure | First snapshot crashes | Try/catch in `flushSnapshotsToStorage`; log + drop snapshot for that session; subsequent flushes retry. |
| Restored buffer wider than new container | Visual truncation/wrap | Webview resizes to snapshot `cols × rows` before write (D8 step 1); `FitAddon` re-fits AFTER content render, producing a clean reflow rather than mid-write corruption. |
| Shell respawn output prints before restore divider | Visual collision | D8 step order: write serialized buffer → write divider → attach DOM → start PTY forwarding. The fresh PTY's prompt appears AFTER the divider. |
